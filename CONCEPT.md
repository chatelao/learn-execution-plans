# CONCEPT.md - Execution Plan Mastery Tutorial

## Goal
The goal of this project is to provide a comprehensive, step-by-step tutorial that enables users to become certifiably knowledgeable about Oracle and PostgreSQL execution plans. The product will bridge the gap between theoretical understanding and practical expertise by providing minimalistic, focused examples alongside complex, real-world scenarios. Each lesson will include installable and executable code samples, paired with interactive exercises to reinforce learning.

## Business Cases
1. **Professional Certification Readiness**: Preparing database administrators and developers for official Oracle and PostgreSQL certification exams by covering all relevant execution plan constructs in depth.
2. **Enterprise Performance Optimization**: Reducing operational costs and improving application responsiveness in corporate environments by training technical staff to identify and resolve query performance bottlenecks independently.
3. **Accelerated Developer Onboarding**: Providing a standardized, hands-on training path for new developers to master database performance engineering, ensuring high-quality, performant code from the start of their tenure.

## Use Cases
1. **Guided Construct Learning**: A user selects a specific execution plan operation (e.g., `INDEX RANGE SCAN` or `NESTED LOOPS`) and follows a guided lesson that explains the construct using a minimalistic, reproducible example.
2. **Interactive Optimization Exercise**: A user is presented with a slow-running query and its execution plan. They must modify the schema or the query to achieve a target execution plan or performance metric, with the system providing immediate feedback.
3. **Full-Context Performance Diagnostics**: A user explores a complex database schema and a set of interconnected queries to understand how multiple execution plan constructs interact in a large-scale system.
4. **Knowledge Verification**: A user completes a series of graded exercises to verify their mastery of execution plan constructs, receiving a summary of their progress and areas for improvement.

## High-Level Functional Architecture
The system is composed of several top-level functional components that interact to provide the learning experience:

1. **Lesson Engine**: Manages the curriculum, tracks user progress, and serves the instructional content (lessons and exercise descriptions).
2. **Database Sandbox Interface**: Provides a secure, isolated environment where users can execute SQL and retrieve real execution plans. It supports multiple database engines (Oracle, PostgreSQL).
3. **Exercise Validator**: Analyzes the execution plans and results generated by the user in the Sandbox to determine if the exercise objectives have been met.
4. **Content Repository**: A structured collection of SQL scripts, schema definitions, lesson text, and metadata for all tutorial modules.

### Business Interfaces
- **Lesson Engine <-> Content Repository**: The Lesson Engine retrieves lesson materials and exercise definitions.
- **Lesson Engine <-> Database Sandbox Interface**: The Lesson Engine triggers the setup of the required database state for a specific lesson or exercise.
- **Database Sandbox Interface <-> User**: The user sends SQL queries and receives execution plans/results.
- **Exercise Validator <-> Database Sandbox Interface**: The Validator requests execution plans from the sandbox to verify user solutions.
- **Lesson Engine <-> Exercise Validator**: The Lesson Engine provides the validation criteria for the current exercise and receives the success/failure status.

## Major Choices

### Choice 1: Database Execution Environment
How will the database engines be provided to the user for running the executable code samples and exercises?

- **Alternative 1: Local Docker Containers**: Provide pre-configured Docker images for Oracle and PostgreSQL that the user runs locally.
  - *Pros*: Full isolation, no internet required after download, real database behavior, no hosting costs.
  - *Cons*: Requires Docker installation and significant local resources (RAM/Disk).
- **Alternative 2: Centralized Cloud-Hosted Database**: Provide a shared, managed database instance in the cloud for all users.
  - *Pros*: Zero local setup, accessible from any device.
  - *Cons*: High operational costs, security risks (user-executed code), difficult to provide isolated states for multiple simultaneous users.
- **Alternative 3: In-Browser SQL Simulator (WASM/Mocking)**: Use a lightweight, simulated SQL engine running in the browser.
  - *Pros*: Instant start, zero setup, very low cost.
  - *Cons*: Extremely difficult to accurately replicate complex execution plan behaviors of Oracle/PostgreSQL; lacks the "real-world" fidelity required for certification.

**Chosen Alternative: Alternative 1 - Local Docker Containers.**
This choice provides the best balance of fidelity, isolation, and cost-effectiveness, ensuring users interact with actual database engines as they would in a professional environment.
